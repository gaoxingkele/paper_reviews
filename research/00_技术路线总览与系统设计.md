## 一、领域共识（一句话结论）

> **当前阶段 AI 审稿应定位为「人类评审的辅助 / 投稿前自检 / 编辑预筛」，由人类对结论负责；价值在"人机互补"——AI 强在技术细节、可复现核查、覆盖度，人类强在新颖性判断与学术品味。**（survey §"共存阶段"；ICLR 2025 "feedback agent"；FactReview/arXiv:2604.04074 主张"审计声明、不做录用决策"）

这决定了我们系统的产品定位：**面向作者「投稿前，按目标期刊要求生成专业、针对性、水平匹配的详细审稿意见与修改建议」**，而非冒充正式同行评审。

## 二、必须遵守的可靠性原则（多来自 AgentReview 的反面实证）

| 原则 | 依据 | 工程落地 |
|---|---|---|
| **先独立、后聚合**：评审 agent 初评阶段相互隔离 | 互见使评分标准差骤降 27.2%，偏见传染 | 维度 reviewer 并行、互不可见；"讨论"是可选开关 |
| **保留分歧，不追求一致** | 讨论制造虚假共识；趋同≠正确 | 上报评审间方差，高分歧→标记"需人工复核" |
| **聚合不独裁也不盲从多数** | inclusive AC 最稳；独裁 κ=0.27 | meta-reviewer 兼听+独立判断，加权而非简单投票 |
| **严格匿名化输入** | 仅 10% 暴露名人作者→决策变 27.7% | 清洗作者/机构/致谢/自引/资助等身份信号 |
| **结构化文字为主、数字为辅** | 去掉数值后决策仅 0.20 重叠（数字被当捷径） | 输出以"证据化理由"为主，分数附带且可校准 |
| **分数校准去偏** | 通用 LLM 系统性偏高（GPT-4o 6.9–8.1）；微调对齐后误差 0.96 vs 2.34 | 加 CalibrationAgent，把 rubric 分映射到目标刊物分布 |
| **多模型投票/并集** | 不同模型抓错互补，并集抓错率 83.3% | 关键维度跨 provider 多次采样，证据并集 |
| **强模型 + 显式激励"挑毛病"** | 弱模型被内容过滤删批评、变"好好先生" | 主评审走 Cloubic 强模型链；prompt 强制列 weaknesses |
| **对抗式核验对抗幻觉** | 幻觉是首要挑战；FactReview 执行代码改变 17% 判定 | VerificationAgent：RAG 引文核查 + (可选)代码/数学执行 |

## 三、采用的审稿流水线（8+ 阶段，融合 survey 管线 + AgentReview 阶段 + DeepReview 三段法）

```
[0] Ingest/Parse      解析稿件（优先 LaTeX 源，保留数学记号；PDF 次之）→ 结构化(标题/摘要/章节/图表/参考文献)
[1] Anonymize         去身份化（作者/机构/致谢/资助/明显自引）
[2] Desk-Screen +     桌面初筛(格式/篇幅/范围/伦理红线) + 期刊匹配度评估(VenueMatch)
    Venue-Match       └─ 输出"与目标刊契合度 + 建议档次"，并据此设定后续严格度
[3] Independent       多维度 reviewer 并行、隔离评审（详见角色表）
    Review            每维度产出: 证据化 strengths/weaknesses/questions + rubric 分
[4] Verification      对抗式核验: 新颖性 RAG(查重/查新) + 经验性 claim 核查 + (可选)代码/数学执行
    (tool-use)        └─ 可推翻或下调前一步的乐观结论
[5] Calibration       分数校准: rubric 分 → 目标刊物分布(线性回归/anchor 样例/多轮均值)
[6] Aggregate +       Meta-Review/AC: inclusive 聚合，保留方差，给整体意见+建议决策+置信度
    Meta-Review       └─ 高分歧/低置信 → 标记需人工复核
[7] (可选) Rebuttal    作者答辩→定向复评（默认关闭；若开启须确保真正影响结论，避免走过场）
[8] Review-Critique    审稿质量自评闭环: specificity/coverage/groundedness 过滤笼统、跑题、无依据的意见
[9] Render            产出: 给作者的《详细审稿意见 + 分维度修改清单》(MD/PDF)
```

并行扩展（DeepReview 测试时算力伸缩）：每个维度可多次采样 + self-verification，再 reduce。

## 四、智能体角色清单（agent 集群）

| Agent | 职责 | 关键技术 | 来源 |
|---|---|---|---|
| **IngestAgent** | 稿件解析为结构化 JSON（章节/图表/公式/引用） | LaTeX>PDF；图表多模态 | survey；arXiv:2505.23824(LaTeX 更稳) |
| **AnonymizerAgent** | 去身份化 | 规则+LLM | AgentReview |
| **DeskScreenAgent** | 范围/格式/伦理红线初筛 | rubric gate | survey(初筛 90%) |
| **VenueMatchAgent** | 评估与目标期刊契合度、建议档次、设定严格度 | 期刊画像 RAG | survey(定制化空白)；分层匹配需求 |
| **维度 Reviewer ×N** | 逐维度独立评审 | 强模型+强制挑刺+证据化 | DeepReviewer 多视角；MAMORX 按维度分 agent |
| ↳ NoveltyReviewer | 新颖性/贡献 | RAG 查新(SchNovel) | survey；MAMORX |
| ↳ SoundnessReviewer | 技术正确性/方法 | reasoning 模型 | DeepReview |
| ↳ ExperimentReviewer | 实验充分性/基线/统计/消融 | claim 核查 | FactReview |
| ↳ ReproReviewer | 可复现性/代码/数据 | (可选)代码执行 | FactReview(执行改 17% 判定) |
| ↳ RelatedWorkReviewer | 文献覆盖/引用恰当 | RAG | MARG |
| ↳ ClarityReviewer | 表述/组织/图表 | — | survey |
| ↳ EthicsReviewer | 伦理/合规/利益冲突 | rubric gate | survey 三层防御 |
| **VerificationAgent** | 对抗式核验(refute 视角)，可推翻乐观结论 | RAG+tool-use+多模型 | FactReview；多模型并集 |
| **CalibrationAgent** | 分数去偏、对齐目标刊分布 | 回归/anchor | OpenReviewer(arXiv:2412.11948) |
| **MetaReviewAgent (AC)** | inclusive 聚合，整体意见+建议决策+置信度+保留分歧 | 加权非独裁 | AgentReview(inclusive AC 最稳) |
| **ReviewCriticAgent** | 审稿质量自评（笼统/跑题/无依据过滤） | ReviewCritique 指标 | survey；ICLR'25 specificity |
| **(可选) AIDetectAgent** | 投稿是否疑似 AI 代写（仅产出"风险提示"，不入分/决策） | Fast-DetectGPT 白盒微服务 | Researcher detector；arXiv:2603.20450(政策不可执行,慎用) |
| **(可选) AuthorAgent / DiscussionAgent** | rebuttal / 评审间讨论 | 默认关闭 | AgentReview(走过场风险) |

> 架构骨架可借鉴 **MARG 的 leader–worker–expert**（worker 按章节切分长文，expert 按维度）与 **ReViewGraph 的"把讨论建成结构化图"** 思路（识别礼貌措辞下的隐性负面共识）。

## 五、期刊配置驱动（本系统的差异化核心 = survey 指出的最大空白）

每个目标期刊一份 `config/journals/<venue>.yaml`，作为一等输入：
- **aims & scope**（用于 VenueMatch + 范围初筛）
- **评审 rubric**：维度清单 + 各维度评分量表（如 IEEE Access 重原创/可复现，不评格式）
- **目标分布**：该刊录用线/分数分布（用于 Calibration 去偏）
- **reviewer persona 组合**：启用哪些维度 agent、各自严格度/权重
- **few-shot 样例**：优秀/被拒样例（anchor 校准 + 风格对齐）
- **政策红线**：AI 使用披露、伦理、篇幅、利益冲突
- **学科/水平档次**：调节严格度，实现"水平匹配期刊要求"

调度器 `orchestrator.py` 读 YAML → 实例化对应 agent 集群 → 跑流水线。换刊物=换 YAML，代码不动。
首个落地样例：用 `D:\BaiduSyncdisk\2026-06\科东论文3篇\审稿要求.md`（IEEE Access 内容向 rubric）生成 `ieee_access.yaml`。

## 六、评测与质量监控（必须自带，否则不可信）

- **与人类一致性**（代理真值=多位人类 reviewer 均分）：MSE/MAE、Spearman、**pairwise 排序准确率**、Decision F1（Researcher `evalate.py` 范式）
- **LLM-as-Judge win-rate**：A/B 随机交换消偏 + 4 维证据化仲裁（Researcher `win_rate_evaluate.py`）
- **可靠性/偏见敏感度**：同稿多配置重跑，测身份/锚定/从众敏感度（AgentReview 方法）
- **审稿意见质量**：specificity / coverage / groundedness / mismatch（ReviewCritique）
- 冷启动数据集：Review-5K、DeepReview-13K、NLPeer、MOPRD、PEERSUM（多源 OpenReview）

## 七、模型与工程

- **基线模型**：复用 news-monitor 的 Cloubic 统一路由（`.env.cloubic`）——claude/gemini/openai/glm/qwen/deepseek 走 Cloubic 降级链；kimi/doubao 直连。已移植 `cloubic.py` + `llm_client.py`。
- **主评审/决策**：强模型 + reasoning 变体（o3/DeepSeek-R1 类）；**多 provider 投票**用于关键维度。
- **输出约束**：用 JSON mode / function calling 强约束结构化输出（规避 Researcher 纯正则解析易崩的坑）。
- **隐私合规**：默认本地/私有部署；稿件不外发非必要服务；输出带"AI 辅助生成"声明。
- **AI 检测**做成独立微服务（需 token logits，OpenAI 兼容 API 拿不到），仅产出风险提示。
